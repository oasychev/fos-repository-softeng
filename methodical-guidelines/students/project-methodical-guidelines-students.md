# МЕТОДИЧЕСКИЕ УКАЗАНИЯ ПО ВЫПОЛНЕНИЮ КУРСОВОГО ПРОЕКТА

## по дисциплине «Разработка прикладных программных решений для систем искусственного интеллекта»

---

### 1. ОБЩИЕ ПОЛОЖЕНИЯ

#### 1.1. Цель курсового проекта

Курсовой проект направлен на формирование системных знаний и практических навыков разработки промышленных программных решений для систем искусственного интеллекта с использованием Python, классических ML-методов и инженерных практик MLOps.

Основная **цель работы** — спроектировать, разработать и развернуть сквозное ИИ-приложение, охватывающее все этапы жизненного цикла: от анализа и обработки данных до развертывания модели в виде микросервиса с мониторингом.

#### 1.2. Задачи курсового проекта

В ходе выполнения курсового проекта студент должен:

1. Научиться проектировать промышленные программные решения (бэкенд-разработка) на основе Python-стека и SQL баз данных.
2. Развить умения и навыки промышленной разработки программных решений (Git, GitFlow, poetry, pytest, pydantic, PEP8).
3. Приобрести навыки реализации ML-моделей на основе библиотек и их включения в программную систему (NumPy, Pandas, Matplotlib, Seaborn, Scikit-learn).
4. Изучить методы и библиотеки разработки внешних интерфейсов системы и ее развертывания в продукционной среде (REST API, FastAPI, Swagger, Docker, CI/CD).

#### 1.3. Место курсового проекта в структуре дисциплины

Курсовой проект выполняется в рамках дисциплины «Разработка прикладных программных решений для систем искусственного интеллекта» и является формой промежуточной аттестации. Работа представляет собой **поэтапную разработку одного проекта**, последовательно охватывающую все разделы курса.

| Этап | Раздел курса | Основные технологии | Результат |
|------|--------------|---------------------|-----------|
| **Этап 1** | Архитектура и информационное обеспечение | Pandas, NumPy, SQL, PostgreSQL, ER-диаграммы | Анализ данных, EDA, структура БД |
| **Этап 2** | Программная инженерия и ML | Scikit-learn, Poetry, pre-commit, pytest | ML-модель, пайплайн, тесты |
| **Этап 3** | Веб-разработка | FastAPI, Pydantic, asyncpg | REST API с интеграцией БД и модели |
| **Этап 4** | Контейнеризация и деплой | Docker, Docker Compose, GitHub, GitHub Actions | Готовый к развертыванию сервис |

1. **Наличие открытых данных** — датасет должен быть доступен для загрузки (Kaggle, UCI, открытые реестры).

### 3. СТРУКТУРА И СОДЕРЖАНИЕ КУРСОВОГО ПРОЕКТА

Курсовой проект выполняется в **четыре этапа**. Каждый этап завершается сдачей отчётных материалов и демонстрацией работающих компонентов.

#### ЭТАП 1. АНАЛИЗ ДАННЫХ И РАЗВЕДОЧНЫЙ АНАЛИЗ (EDA)

**Цель этапа:** Освоить инструменты анализа данных (Pandas, NumPy), визуализации (Matplotlib, Seaborn) и проектирования структуры базы данных (SQL, PostgreSQL) для последующего использования в ИИ-приложении.

##### 1.1. Выбор и загрузка данных

- Выбрать датасет из открытых источников.
- Загрузить данные в Jupyter Notebook или Python-скрипт.
- Изучить структуру данных: количество строк, столбцов, типы данных.

##### 1.2. Предварительная обработка данных (Pandas, NumPy)

- Выполнить первичный осмотр данных: вывести первые 5 строк, общую информацию (`df.info()`), описательные статистики (`df.describe()`).
- Проверить наличие пропусков и выбрать стратегию их заполнения (среднее, медиана, мода или удаление).
- Проверить наличие дубликатов и удалить их при необходимости.
- Преобразовать типы данных для оптимизации памяти (использовать параметр `dtype`).
- Выполнить кодирование категориальных признаков (`pd.get_dummies` или `LabelEncoder`).
- Масштабировать числовые признаки с использованием `StandardScaler` или `MinMaxScaler`.

##### 1.3. Разведочный анализ данных (EDA)

- Построить гистограммы для всех числовых признаков.
- Построить столбчатые диаграммы для категориальных признаков.
- Построить корреляционную матрицу (тепловая карта с Seaborn) и проанализировать корреляции признаков с целевой переменной.
- Проверить сбалансированность классов (распределение целевой переменной).
- Сформулировать **не менее 3 содержательных гипотез** о факторах, влияющих на целевую переменную, и подтвердить или опровергнуть их с помощью данных.

##### 1.4. Проектирование базы данных

- Выделить сущности на основе данных.
- Построить ER-диаграмму (использовать draw.io, Lucidchart или нарисовать вручную).
- Определить первичные и внешние ключи.
- Привести таблицы к 3-й нормальной форме (3NF).
- Создать SQL-скрипты для создания таблиц в PostgreSQL с учётом нормализации.

##### 1.5. Документирование и инженерные практики

- Оформить код в формате Jupyter Notebook или Python-скриптов.
- Добавить `type hints` для всех функций.
- Добавить `docstrings` в Google-стиле.
- Настроить `pre-commit` хуки: `trailing-whitespace`, `end-of-file-fixer`, `ruff`.
- Подготовить `README.md` с описанием структуры проекта, данных и инструкцией по запуску.

##### Результат этапа 1

- Jupyter Notebook с EDA (или Python-скрипты с комментариями).
- ER-диаграмма и SQL-скрипты для создания таблиц.
- Отчёт по EDA с гипотезами и выводами (в формате Markdown или PDF).
- Проект с настроенным Poetry, pre-commit и базовой структурой.
- **Тег в репозитории:** `v1.0`

#### ЭТАП 2. РАЗРАБОТКА ML-МОДЕЛИ И ПАЙПЛАЙНА

**Цель этапа:** Разработать модель машинного обучения для прогнозирования, построить ETL-пайплайн для загрузки данных в БД и оценить качество модели.

##### 2.1. ETL-пайплайн (PostgreSQL + Python)

- Создать базу данных в PostgreSQL (локально или в Docker-контейнере).
- Разработать скрипт для загрузки данных из исходного CSV-файла в нормализованную БД.
- Использовать `psycopg2` или `asyncpg` для подключения к БД.
- Реализовать пакетную вставку данных с использованием `COPY` для оптимизации.
- Написать SQL-запросы для извлечения агрегированных данных (например, количество услуг на клиента, средний платёж и т.д.) с использованием `GROUP BY`, `JOIN`, оконных функций.

##### 2.2. Разработка ML-модели (Scikit-learn)

- Разделить данные на обучающую и тестовую выборки (70/30 или 80/20) с учётом стратификации.
- Обучить **минимум 3 модели** для решения задачи (бинарной классификации, регрессии или кластеризации):
  - Логистическая регрессия (базовая модель) — для классификации.
  - Случайный лес (Random Forest).
  - Градиентный бустинг (XGBoost, LightGBM или CatBoost).
- Провести подбор гиперпараметров (GridSearchCV или RandomizedSearchCV).
- Выполнить оценку качества моделей с использованием соответствующих метрик:
  - Для классификации: точность (Accuracy), полнота (Recall), точность (Precision), F1-мера, ROC-AUC.
  - Для регрессии: MSE, MAE, R².
- Сравнить модели между собой и выбрать лучшую.

##### 2.3. Построение пайплайна предобработки

- Создать полноценный пайплайн с использованием `sklearn.pipeline.Pipeline`, включающий:
  - Стандартизацию числовых признаков.
  - Кодирование категориальных признаков.
  - Выбор лучших признаков (опционально).
- Сохранить обученный пайплайн и модель с помощью `joblib` или `pickle`.

##### 2.4. Логирование и тестирование

- Настроить логирование (модуль `logging`) всех этапов обучения и обработки данных.
- Написать модульные тесты (pytest) для:
  - Проверки работы ETL-скриптов.
  - Проверки загрузки и валидации данных.
  - Проверки пайплайна предобработки.
- Добиться покрытия тестами **≥80%**.

##### 2.5. Документирование и инженерные практики

- Продолжить использование `type hints` и `docstrings`.
- Интегрировать запуск тестов в `pre-commit`.
- Добавить конфигурацию через `.env` (параметры подключения к БД, пути к данным).

##### Результат этапа 2

- Обученная ML-модель и пайплайн предобработки (сохранённые в файлы).
- ETL-скрипты для загрузки данных в PostgreSQL.
- Отчёт с результатами сравнения моделей и выбранными гиперпараметрами.
- Тесты и логи работы пайплайна.
- **Тег в репозитории:** `v2.0`

#### ЭТАП 3. РАЗРАБОТКА REST API И ИНТЕГРАЦИЯ С БД

**Цель этапа:** Разработать веб-сервис на FastAPI с эндпоинтами для получения прогнозов, управления данными и получения метрик качества, с интеграцией с PostgreSQL и загруженной ML-моделью.

##### 3.1. Разработка API на FastAPI

- Создать приложение FastAPI с базовой структурой (роутеры, схемы Pydantic).
- Реализовать **минимум 4 эндпоинта**:

| Эндпоинт | Метод | Назначение |
|----------|-------|------------|
| `/predict` | POST | Принимает данные о клиенте (в формате JSON), возвращает прогноз и вероятность. |
| `/health` | GET | Проверка работоспособности сервиса. |
| `/metrics` | GET | Возвращает метрики качества модели на тестовых данных. |
| `/features` | GET | Возвращает список признаков, используемых моделью, с описанием. |
| `/history` | GET (опционально) | Возвращает историю прогнозов (с пагинацией). |

##### 3.2. Валидация данных (Pydantic)

- Создать Pydantic-схемы для валидации входных данных на эндпоинте `/predict`.
- Добавить валидацию типов, диапазонов значений и обязательных полей.
- Обрабатывать ошибки валидации и возвращать понятные сообщения пользователю.

##### 3.3. Асинхронная работа с БД

- Использовать `asyncpg` для асинхронного подключения к PostgreSQL.
- Реализовать асинхронный пул соединений.
- Написать асинхронные функции для:
  - Сохранения запросов и ответов в таблицу `predictions_history`.
  - Получения статистики по истории прогнозов.
  - Получения данных о клиентах из БД при необходимости.

##### 3.4. Логирование и обработка ошибок

- Настроить логирование запросов, ошибок и времени выполнения эндпоинтов.
- Реализовать кастомные исключения и глобальный обработчик ошибок.
- Логировать каждое обращение к `/predict` с идентификатором запроса.

##### 3.5. Документация API

- Использовать встроенную документацию Swagger (`/docs`) и ReDoc (`/redoc`).
- Добавить описания для всех эндпоинтов, примеры запросов и ответов.
- Подготовить OpenAPI-спецификацию в формате JSON.

##### 3.6. Тестирование

- Написать интеграционные тесты для эндпоинтов с использованием `pytest` и `httpx`.
- Написать тесты для асинхронных запросов к БД.
- Использовать `pytest-mock` для мокирования внешних зависимостей (например, ML-модели).
- Добиться покрытия кода API **≥85%**.

##### Результат этапа 3

- FastAPI-приложение с реализованными эндпоинтами.
- Pydantic-схемы для валидации.
- Асинхронное подключение к PostgreSQL.
- Swagger-документация API.
- Тесты и логи.
- **Тег в репозитории:** `v3.0`

#### ЭТАП 4. КОНТЕЙНЕРИЗАЦИЯ, ДЕПЛОЙ И МОНИТОРИНГ

**Цель этапа:** Контейнеризовать разработанное приложение с использованием Docker, настроить многоконтейнерное развертывание через Docker Compose и организовать непрерывную интеграцию через GitHub Actions.

##### 4.1. Контейнеризация приложения (Docker)

- Создать `Dockerfile` для FastAPI-приложения:
  - Использовать официальный Python-образ (`python:3.10-slim`).
  - Установить зависимости из Poetry (`poetry install`).
  - Скопировать код и модель.
  - Определить команду запуска (`CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]`).
- Создать `Dockerfile` для PostgreSQL (или использовать готовый образ).
- Оптимизировать размер образа (многослойная сборка, использование `--no-cache-dir`).

##### 4.2. Многоконтейнерное развертывание (Docker Compose)

- Создать `docker-compose.yml` для запуска трёх сервисов:
  - **app** — FastAPI-приложение.
  - **db** — PostgreSQL.
  - **pgadmin** (опционально) — для управления БД.
- Настроить переменные окружения через `.env` для всех сервисов.
- Настроить сети и тома (volumes) для сохранения данных БД.
- Убедиться, что приложение ожидает запуска БД (использовать `depends_on` или скрипт `wait-for-it.sh`).

##### 4.3. CI/CD через GitHub Actions

- Настроить GitHub Actions для автоматической сборки и тестирования приложения при каждом push-е:
  - Проверка форматирования (ruff).
  - Запуск тестов (pytest) в контейнере.
  - Сборка Docker-образа.
  - Публикация образа в Docker Hub (или GitHub Container Registry).
- Создать пайплайн для автоматического деплоя на сервер (опционально).

##### 4.4. Мониторинг и наблюдаемость

- Интегрировать логирование в JSON-формате (для совместимости с ELK или Loki).
- Добавить эндпоинт `/metrics` с Prometheus-метриками (количество запросов, время ответа, ошибки) — использовать `prometheus-client`.
- Настроить сбор метрик и визуализацию в Grafana (опционально, для продвинутого уровня).

##### 4.5. Документирование и финальная проверка

  - Инструкции по установке через Poetry.

- Заключение с выводами и перспективами развития.
- Список использованных источников.

### 9. РЕКОМЕНДАЦИИ ПО ВЫПОЛНЕНИЮ

#### 9.1. Организация работы

1. **Ведите репозиторий с самого начала.** Создайте репозиторий на GitHub и настройте Poetry, pre-commit и базовую структуру проекта до начала выполнения этапов.
2. **Используйте теги для этапов.** После завершения каждого этапа создавайте тег в репозитории (`v1.0`, `v2.0` и т.д.).
3. **Ведите документацию параллельно с разработкой.** Не откладывайте написание README и отчёта на последний момент.
4. **Регулярно делайте коммиты.** Каждый коммит должен иметь осмысленное сообщение и соответствовать одному логическому изменению.

#### 9.2. Работа с данными

1. **Начинайте с EDA в Jupyter Notebook.** Это позволит быстро экспериментировать и визуализировать результаты.
2. **Фиксируйте все выводы и гипотезы.** Они пригодятся при написании отчёта и обосновании выбора модели.
3. **Проверяйте сбалансированность классов.** Для несбалансированных данных используйте соответствующие метрики (F1, ROC-AUC) и техники борьбы с дисбалансом.

#### 9.3. Разработка ML-модели

1. **Начинайте с простой модели** (например, логистическая регрессия) для получения baseline.
2. **Используйте пайплайны** для предобработки — это обеспечит воспроизводимость и упростит интеграцию в API.
3. **Сохраняйте модель** с помощью `joblib.dump()` после каждого удачного эксперимента.
4. **Документируйте эксперименты** — фиксируйте гиперпараметры и полученные метрики.

#### 9.4. Разработка API

1. **Начинайте с реализации эндпоинта `/health`** — он самый простой и позволит проверить работоспособность приложения.
2. **Используйте Pydantic для валидации** — это значительно упростит обработку ошибок.
3. **Пишите тесты параллельно с разработкой** — это поможет быстрее находить и исправлять ошибки.
4. **Используйте асинхронные функции** для работы с БД — это повысит производительность API.

#### 9.5. Контейнеризация и деплой

1. **Настройте Docker Compose для локальной разработки** — это упростит тестирование интеграции всех компонентов.
2. **Используйте `.env` для конфигурации** — не храните секреты в коде.
3. **Настройте GitHub Actions на раннем этапе** — это позволит автоматически проверять качество кода при каждом push-е.

#### 9.6. Частые ошибки

1. **Пропуск этапа нормализации БД** — приводит к дублированию данных и проблемам с производительностью.
2. **Отсутствие тестов** — значительно усложняет отладку и снижает качество кода.
3. **Игнорирование type hints и docstrings** — снижает читаемость и усложняет сопровождение кода.
4. **Хранение модели в репозитории** — модель может иметь большой размер; используйте Git LFS или загружайте отдельно.
5. **Отсутствие обработки ошибок в API** — приводит к неинформативным ответам при некорректных запросах.
