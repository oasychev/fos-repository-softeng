# Методические указания по выполнению лабораторных работ

## Модуль 4. Инструменты промышленной разработки на Python


### 1. Общие положения

Настоящие методические указания предназначены для студентов 2-го курса, изучающих модуль «Инструменты промышленной разработки на Python». Лабораторные работы направлены на освоение современного инструментария промышленной разработки: управления зависимостями через Poetry, автоматизации контроля качества кода с pre-commit, написания модульных тестов с pytest, организации логирования, строгой типизации и документирования кода, а также управления конфигурациями с использованием Pydantic и python-dotenv.

Лабораторные работы выполняются в два этапа:
- **Аудиторная часть (4 часа)** — выполняется под руководством преподавателя в аудитории.
- **Самостоятельная работа** — выполняется студентом самостоятельно после аудиторных занятий.

Защита работы включает демонстрацию работоспособности всех заданий аудиторной и самостоятельной частей, а также устные ответы на теоретические и практические вопросы.


### 2. Общие требования ко всем вариантам

#### 2.1. Требования к инструментарию

При выполнении всех вариантов лабораторных работ необходимо использовать следующий стек инструментов:

| Инструмент | Назначение |
|------------|------------|
| **Poetry** | Управление зависимостями, создание виртуального окружения, упаковка проекта |
| **Git** | Контроль версий |
| **pre-commit** | Автоматическая проверка качества кода перед коммитом |
| **pytest + pytest-cov + pytest-mock** | Модульное тестирование и оценка покрытия |
| **logging** | Логирование |
| **Pydantic + pydantic-settings** | Валидация данных и управление конфигурацией |
| **python-dotenv** | Загрузка переменных окружения из `.env`-файла |
| **Type Hints** | Статическая типизация |
| **Docstrings (Google-style)** | Документирование кода |
| **Ruff** | Линтинг и форматирование (в составе pre-commit) |

#### 2.2. Требования к качеству кода

1. **Type Hints**: все функции и методы должны содержать аннотации типов для аргументов и возвращаемых значений.
2. **Docstrings**: все публичные функции, классы и модули должны быть документированы в Google-стиле.
3. **PEP8**: код должен соответствовать стандарту PEP8.
4. **Pre-commit**: должны быть настроены хуки `trailing-whitespace`, `end-of-file-fixer`, `ruff`.

#### 2.3. Требования к логированию

Во всех вариантах работ необходимо настроить логирование с выводом в консоль и в файл. Логированию подлежат:
- Начало и завершение ключевых операций (запросов, парсинга, сохранения);
- Ошибки и исключения;
- Уровень логирования должен управляться через конфигурацию (`.env` или аргументы CLI).

#### 2.4. Требования к тестированию

- Модульные тесты должны покрывать не менее **85%** кода (`pytest --cov`).
- Для тестирования HTTP-запросов и внешних зависимостей использовать `pytest-mock`.
- Тесты должны запускаться автоматически через pre-commit.


### 3. Вариант 1. Консольный анализатор погодных данных

#### 3.1. Цель работы

Разработать консольное приложение для получения, валидации и отображения данных о погоде из публичного API OpenWeatherMap.

#### 3.2. Аудиторная часть (4 часа)

##### Этап 1. Инициализация проекта (Poetry)

1. Создать проект с помощью Poetry:
   ```bash
   poetry new weather_analyzer
   cd weather_analyzer
   ```

2. Добавить основные зависимости:
   ```bash
   poetry add requests pydantic pydantic-settings python-dotenv
   ```

3. Добавить dev-зависимости:
   ```bash
   poetry add --group dev pytest pytest-cov pytest-mock pre-commit
   ```

##### Этап 2. Создание структуры проекта

Создать следующую структуру модулей:

```
weather_analyzer/
├── weather_analyzer/
│   ├── __init__.py
│   ├── models.py      # Pydantic-модель WeatherRecord
│   ├── fetcher.py     # Функция fetch_weather()
│   ├── config.py      # Класс Settings
│   └── cli.py         # Точка входа с argparse
├── tests/
│   └── __init__.py
├── pyproject.toml
└── README.md
```

**Модуль `models.py`**:
- Создать Pydantic-модель `WeatherRecord` с полями:
  - `city: str`
  - `temperature: float`
  - `humidity: int`
  - `pressure: int`
  - `timestamp: datetime`

**Модуль `config.py`**:
- Создать класс `Settings`, наследующий от `BaseSettings`;
- Определить переменные: `API_KEY`, `BASE_URL`, `LOG_LEVEL`, `TIMEOUT`;
- Настроить загрузку из `.env`-файла.

**Модуль `fetcher.py`**:
- Реализовать функцию `fetch_weather(city: str) -> dict`, выполняющую GET-запрос к OpenWeatherMap API;
- Обработать возможные ошибки (неверный город, таймаут, сетевые ошибки).

**Модуль `cli.py`**:
- Использовать `argparse` для приёма названия города;
- Вызвать `fetch_weather()` и вывести результат в консоль;
- Команда запуска: `python -m weather_analyzer.cli Moscow`.

##### Этап 3. Настройка pre-commit

1. Создать файл `.pre-commit-config.yaml` в корне проекта:
   ```yaml
   repos:
     - repo: https://github.com/pre-commit/pre-commit-hooks
       rev: v4.5.0
       hooks:
         - id: trailing-whitespace
         - id: end-of-file-fixer
     - repo: https://github.com/astral-sh/ruff-pre-commit
       rev: v0.3.0
       hooks:
         - id: ruff
           args: [--fix]
   ```

2. Установить хуки:
   ```bash
   pre-commit install
   ```

3. Проверить работу:
   ```bash
   pre-commit run --all-files
   ```

##### Этап 4. Настройка логирования

1. Настроить логгер с выводом в консоль и в файл `weather.log`;
2. Логировать:
   - Начало запроса к API;
   - Успешное завершение запроса;
   - Ошибки (неверный город, таймаут и др.).

##### Результат аудиторной части

К концу занятия приложение должно работать по команде:
```bash
python -m weather_analyzer.cli Moscow
```
и выводить в консоль текущую погоду для указанного города.

#### 3.3. Самостоятельная работа

##### Задание 1. Модульные тесты

Написать тесты с использованием `pytest` и `pytest-mock`:

1. **Тест успешного получения данных**:
   - Мокировать `requests.get()` для возврата корректного JSON-ответа;
   - Проверить, что `fetch_weather()` возвращает ожидаемый словарь.

2. **Тест обработки ошибок**:
   - Неверный город (404) — проверить, что выбрасывается исключение;
   - Таймаут — проверить обработку `requests.exceptions.Timeout`.

3. **Тест валидации модели**:
   - Проверить, что Pydantic-модель `WeatherRecord` валидирует некорректные значения (отрицательная влажность, отсутствие обязательных полей).

Пример структуры теста с моком:
```python
def test_fetch_weather_success(mocker):
    mock_response = mocker.Mock()
    mock_response.json.return_value = {...}
    mock_response.raise_for_status.return_value = None
    mocker.patch('requests.get', return_value=mock_response)
    
    result = fetch_weather("Moscow")
    assert result["city"] == "Moscow"
```

##### Задание 2. Поддержка .env

1. Создать файл `.env` в корне проекта:
   ```
   API_KEY=your_api_key_here
   BASE_URL=https://api.openweathermap.org/data/2.5
   LOG_LEVEL=DEBUG
   TIMEOUT=10
   ```

2. Проверить работу приложения с разными уровнями логирования (DEBUG, INFO, ERROR).

##### Задание 3. Покрытие тестами

1. Запустить тесты с оценкой покрытия:
   ```bash
   poetry run pytest --cov=weather_analyzer --cov-report=term-missing
   ```

2. Добиться покрытия **≥85%**.

##### Задание 4. Интеграция тестов в pre-commit

Добавить в `.pre-commit-config.yaml` хук для запуска тестов:
```yaml
- repo: local
  hooks:
    - id: pytest
      name: pytest
      entry: poetry run pytest
      language: system
      types: [python]
      pass_filenames: false
```

##### Задание 5. README.md

Подготовить файл `README.md` с инструкцией:
- Установка и настройка проекта;
- Создание `.env`-файла с API-ключом;
- Запуск приложения;
- Запуск тестов.


### 4. Вариант 2. Парсер и валидатор данных с веб-страниц

#### 4.1. Цель работы

```bash

poetry add requests beautifulsoup4 pydantic pydantic-settings python-dotenv
```

##### Этап 2. Создание структуры проекта

```
html_parser/
├── html_parser/
│   ├── __init__.py
│   ├── models.py      # Pydantic-модель BookRecord
│   ├── parser.py      # Функция parse_page()
│   ├── saver.py       # Функция save_to_json()

│   └── cli.py         # Точка входа с argparse
├── tests/
├── pyproject.toml
└── README.md
```

**Модуль `models.py`**:
- Создать Pydantic-модель `BookRecord` с полями:
  - `title: str`
  - `author: str` (может отсутствовать — использовать `Optional`)
  - `price: float`
  - `rating: int` (1–5)
  - `availability: bool`

**Модуль `parser.py`**:
- Реализовать функцию `parse_page(url: str) -> list[dict]`, использующую `BeautifulSoup`;

- Для каждой книги извлечь название, цену, рейтинг и наличие.

**Модуль `saver.py`**:
- Реализовать функцию `save_to_json(data: list[BookRecord], filename: str)`;
- Сохранять данные в формате JSON.

**Модуль `cli.py`**:
- Использовать `argparse` для приёма аргументов `--url` и `--output`;

### 6. Критерии оценивания

| Критерий | Баллы |
|----------|-------|
| Выполнение аудиторной части задания | 0–1 |
| Выполнение самостоятельной части задания | 0–2 |
| Защита работы (демонстрация + ответы на вопросы) | 0–4 |
| **Итого** | **0–7** |

#### 6.1. Критерии оценки аудиторной части (0–1 балл)

- **1 балл**: все задания аудиторной части выполнены в полном объёме, код работает, структура проекта соответствует требованиям.
- **0 баллов**: задания выполнены не в полном объёме или код не работает.

#### 6.2. Критерии оценки самостоятельной части (0–2 балла)

- **2 балла**: все задания самостоятельной работы выполнены, покрытие тестами ≥85%, pre-commit настроен, README подготовлен.
- **1 балл**: задания выполнены частично, покрытие тестами 50–84%.
- **0 баллов**: задания не выполнены или покрытие тестами <50%.

#### 6.3. Критерии защиты работы (0–4 балла)

- **4 балла**: студент демонстрирует работу всех функций, уверенно отвечает на теоретические и практические вопросы, показывает понимание всех использованных инструментов.
- **3 балла**: студент демонстрирует работу всех функций, но испытывает затруднения при ответе на некоторые вопросы.
- **2 балла**: студент демонстрирует работу только базовых функций, отвечает на вопросы с ошибками.
- **1 балл**: студент не может продемонстрировать работу приложения, но отвечает на отдельные вопросы.
- **0 баллов**: работа не представлена.
