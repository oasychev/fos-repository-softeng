# Методические указания для преподавателя

## Модуль 4. Инструменты промышленной разработки на Python

### Лабораторные работы (КИМ-4.2)


### 1. Цели и задачи лабораторных работ

#### 1.1. Цель лабораторных работ
Сформировать у студентов практические навыки использования современного инструментария промышленной разработки на Python (Poetry, pre‑commit, pytest, logging, Pydantic, python‑dotenv, type hints, docstrings) через реализацию сквозного проекта от инициализации репозитория до готового к публикации приложения.

#### 1.2. Проверяемые результаты обучения
- **PL-1.1 (Базовый)** — разрабатывает и отлаживает прикладные решения с использованием языка программирования Python, тестирует, испытывает и оценивает качество таких решений.
- **LC-5.1 (Базовый)** — осуществляет выбор инструментов и инженерных практик промышленной разработки систем ИИ, развертывания и сопровождения моделей МО в продуктивной среде.

#### 1.3. Задачи преподавателя
- Обеспечить освоение студентами инструментов и практик, перечисленных выше.
- Организовать аудиторную работу (4 часа) таким образом, чтобы каждый студент к концу занятия имел работающее базовое приложение.
- Контролировать выполнение самостоятельной части и оценивать защиту.
- Формировать у студентов понимание взаимосвязи между инструментами и их ролью в промышленной разработке.

---

### 2. Организация проведения лабораторных работ

#### 2.1. Общая структура

| Этап | Форма работы | Время |
|------|--------------|-------|
| Вводная часть | Лекция-инструктаж (объяснение задания, демонстрация инструментов) | 0,5 ч |
| Аудиторная работа | Самостоятельное выполнение заданий под контролем преподавателя | 3,5 ч |
| Самостоятельная работа | Домашнее выполнение дополнительных заданий | ~8–10 ч |
| Защита | Индивидуальное собеседование с демонстрацией кода | 15–20 мин на студента |

#### 2.2. Варианты заданий
Студенты могут выбирать один из трёх вариантов (или назначаются преподавателем):
1. **Консольный анализатор погодных данных** (OpenWeatherMap).
2. **Парсер и валидатор данных с веб-страниц** (Books to Scrape).
3. **Модуль конфигурации для микросервисов** (множественные источники).

Все варианты равноценны по сложности и охватывают одинаковый набор инструментов. Рекомендуется распределять студентов так, чтобы в группе были представлены все варианты для разнообразия при защите.

#### 2.3. Инфраструктурное обеспечение
- Компьютерный класс с установленными Python ≥3.10, Poetry, Git.
- Доступ в интернет (для установки пакетов и работы с API/сайтами).
- Для варианта 1 преподавателю необходимо заранее получить API-ключ OpenWeatherMap (бесплатный) и предоставить его студентам (или дать инструкцию по получению). Для варианта 2 доступ к books.toscrape.com не требует ключей.
- Рекомендуется иметь готовый репозиторий-шаблон (или примеры конфигурационных файлов) для ускорения начала работы.

### 3. Рекомендации по проведению аудиторной части (4 часа)

#### 3.1. Вводная часть (0,5 ч)
- **Объяснить** цели работы и её связь с реальными проектами.
- **Продемонстрировать** на живом примере:
  - Создание проекта через `poetry new`.
  - Добавление зависимостей.
  - Настройку `pre-commit` с базовыми хуками.
  - Простейший пример использования Pydantic и логирования.
- Раздать студентам **чек-лист** задач на аудиторную часть (см. приложение 1).
- Обратить внимание на требования к оформлению кода (type hints, docstrings).

#### 3.2. Основная часть (3,5 ч)
Студенты работают самостоятельно, преподаватель консультирует, помогает устранять ошибки, проверяет промежуточные результаты.

**Контрольные точки для преподавателя:**

| Время (от начала) | Что должно быть сделано | Действия преподавателя |
|-------------------|-------------------------|------------------------|
| 1 час | Инициализирован проект, добавлены зависимости, создана структура модулей (models, config, fetcher/parser/loader, cli). | Проверить наличие файлов и корректность `pyproject.toml`. |
| 2 часа | Реализованы модели Pydantic, настроен класс Settings с загрузкой из `.env`. Написана базовая логика (запрос к API / парсинг / загрузка). | Просмотреть код моделей и настроек, указать на возможные ошибки валидации. |
| 3 часа | Реализована точка входа (CLI), настроено логирование в файл и консоль. Приложение запускается и выдаёт результат. | Проверить работоспособность, логирование ошибок. |
| 3,5 часа | Добавлены type hints и docstrings для всех функций. Настроен pre-commit (хуки установлены, `pre-commit run --all-files` не выдаёт ошибок). | Проверить выполнение pre-commit, качество документации. |

**Рекомендации по консультированию:**
- Обращать внимание студентов на использование `pydantic-settings` для управления конфигурацией, а не ручной парсинг `.env`.
- Напоминать о необходимости обработки исключений при сетевых запросах (варианты 1, 2) и отсутствующих файлах (вариант 3).
- Подсказывать правильные селекторы BeautifulSoup для варианта 2 (например, `article.product_pod`, `h3 > a`, `p.price_color` и т.д.).
- Поощрять использование `pathlib` для путей к файлам логов и вывода.

#### 3.3. Завершение аудиторной части
- Попросить студентов закоммитить текущее состояние с осмысленным сообщением (например, `feat: base implementation`).
- Проверить, что у каждого студента есть рабочий код, который можно продолжить дома.
- Выдать задание на самостоятельную работу (можно в электронном виде).

### 4. Рекомендации по организации самостоятельной работы

#### 4.1. Содержание самостоятельной работы
Для каждого варианта предусмотрены следующие общие блоки:
1. **Модульное тестирование** (pytest, pytest-mock, pytest-cov).
2. **Доработка конфигурации** (поддержка `.env`, управление уровнями логирования, префиксы).
3. **Расширение функциональности** (пагинация, секреты, пример использования).
4. **Интеграция тестов в pre-commit**.
5. **Подготовка README.md**.

#### 4.2. Рекомендации для преподавателя по контролю
- **Срок сдачи** самостоятельной части — за 2–3 дня до защиты, чтобы успеть проверить код и дать обратную связь.
- Запросить у студентов ссылку на репозиторий (GitHub/GitLab) или архив с проектом.
- Проверить:
  - Наличие всех модулей и файлов.
  - Прохождение всех тестов (`pytest`).
  - Покрытие тестами (`pytest --cov`).
  - Настроенный `pre-commit` (можно проверить, запустив `pre-commit run --all-files`).
  - Наличие `README.md` с инструкцией по установке и запуску.
- Обратить внимание, что тесты должны использовать моки для внешних запросов (не допускать реальных вызовов API в тестах).

#### 4.3. Типичные ошибки в самостоятельной работе
- Отсутствие обработки ошибок при валидации Pydantic (например, не используются `Optional` для необязательных полей).
- Использование реальных сетевых вызовов в тестах вместо моков.
- Неправильная настройка `pytest-cov` (не видит модули).
- Пропуск `__init__.py` в тестовых директориях.
- Игнорирование логирования в самостоятельной части.
- Неправильное форматирование `pyproject.toml` при добавлении скриптов.

### 5. Рекомендации по проведению защиты

#### 5.1. Формат защиты
- Индивидуальное собеседование (15–20 минут на студента).
- Студент демонстрирует работу приложения на примерах (запуск команд, проверка вывода).
- Отвечает на теоретические и практические вопросы по использованным инструментам.

#### 5.2. Примерные вопросы для защиты
**Общие:**
- Как Poetry управляет зависимостями? Чем отличается `dependencies` от `dev-dependencies`?
- Как работает `pre-commit`? Какие хуки вы использовали и зачем?
- В чём преимущества использования Pydantic перед обычными классами?
- Как настроить логирование с разными уровнями? Как оно помогает при отладке?

**По варианту 1:**
- Как вы обрабатывали ошибки API (неверный город, таймаут)?
- Как вы валидировали данные из внешнего API через Pydantic?
- Как бы вы расширили приложение для получения прогноза на несколько дней?

**По варианту 2:**
- Как вы извлекали данные с помощью BeautifulSoup? Какие селекторы использовали?
- Как вы обрабатывали отсутствие автора?
- Как реализовали пагинацию?

**По варианту 3:**
- Каков приоритет источников конфигурации в вашем решении?
- Как вы скрывали секреты при выводе?
- Как бы вы добавили поддержку ещё одного формата (например, TOML)?

#### 5.3. Критерии оценки защиты (0–4 балла)
- **4 балла**: студент уверенно демонстрирует все функции приложения, показывает глубокое понимание каждого инструмента, отвечает на дополнительные вопросы, код соответствует всем требованиям (тесты, документирование, pre-commit).
- **3 балла**: все функции работают, но студент допускает неточности в объяснениях; мелкие замечания по коду.
- **2 балла**: базовая функциональность работает, но есть пробелы в самостоятельной части (например, не все тесты написаны, покрытие <85%) или слабое понимание теории.
- **1 балл**: приложение работает с ошибками или студент не может объяснить свой код.
- **0 баллов**: работа не представлена.

### 9. Приложение 2. Рекомендации по проверке кода

При проверке самостоятельной работы рекомендуется обращать внимание на:

1. **Структура проекта** – соответствует ли шаблону Poetry (наличие `src/` или пакета в корне).
2. **Использование Pydantic** – все ли данные валидируются, используются ли `Field` для описания ограничений.
3. **Конфигурация** – используется ли `pydantic-settings` для загрузки из `.env`, есть ли поддержка разных форматов (для варианта 3).
4. **Логирование** – настроено ли логирование в файл, указаны ли уровни, логируются ли ошибки.
5. **Тесты** – покрытие ≥85%, использование моков, наличие тестов на ошибки.
6. **Pre-commit** – есть ли файл `.pre-commit-config.yaml`, проходят ли хуки без ошибок.
7. **README** – содержит ли инструкции по установке, настройке и запуску.

### 10. Заключительные замечания

Данные лабораторные работы являются важной частью модуля, поскольку они закладывают фундамент для дальнейшей профессиональной деятельности студентов в области разработки ИИ-систем. Акцент следует делать не только на написании кода, но и на понимании *почему* важны те или иные практики (тестирование, контроль качества, конфигурирование). Преподавателю рекомендуется связывать каждое задание с реальными сценариями из индустрии.

Успешное выполнение лабораторных работ и защита гарантируют, что студент сможет самостоятельно создавать поддерживаемые, тестируемые и готовые к развертыванию проекты на Python.
