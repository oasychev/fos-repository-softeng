# Методические указания для преподавателя по оцениванию устных докладов (КИМ-3.1)
## Тема: «Особенности работы с системой контроля версий Git»


## 1. Общие положения и цели контроля

**Цель КИМ-3.1:** Проверка умения студента самостоятельно собирать, систематизировать и критически оценивать профессиональную техническую информацию из различных источников (включая неофициальные), а также способности аргументированно представлять и защищать свои находки в профессиональной дискуссии.

**Формат проведения:** 
- Выступление студента: 2–3 минуты.
- Место: лекционное или практическое занятие.
- Завершающий этап: обязательная дискуссия с преподавателем и группой (уточнение, дополнение, коррекция информации).

**Важно:** Студентам разрешено использование ИИ-ассистентов при подготовке. Задача преподавателя – в ходе дискуссии верифицировать, что студент не просто зачитал сгенерированный текст, а действительно понимает суть описываемых процессов, случаи использования команд, и готов применить их в профессиональной деятельности.

---

## 2. Ожидаемая структура доклада студента

Преподаватель должен контролировать наличие в выступлении следующих четырех блоков:
1. **Назначение команды** (кратко, без зачитывания мануала).
2. **Случаи использования (Use Cases)** – *ключевой блок*. Когда и зачем команда применяется на практике.
3. **Ключевые параметры и их смысл** (основные флаги, аргументы).
4. **Особенности поведения и «подводные камни»** (риски, влияние на историю репозитория, нюансы командной работы).

---

## 3. Детализированные критерии оценивания (Рубрикатор)

Максимальная оценка за доклад составляет **2 балла**. Оценка выставляется по двум независимым критериям (по 1 баллу каждый).

### Критерий 1: Полнота покрытия случаев использования (Use Cases) – 1 балл
| Оценка | Дескрипторы (что наблюдает преподаватель) |
| :--- | :--- |
| **1 балл** | Студент привел 2 и более конкретных, реалистичных сценария использования команды в профессиональной разработке (например, в контексте ML Ops, командной работы, hotfix). Объяснено, *какую проблему* решает команда в каждом сценарии. Информация выходит за рамки тривиальных примеров официальной документации. |
| **0 баллов** | Случаи использования не названы, ограничены одним абстрактным утверждением («чтобы сохранить изменения»), или сценарии явно выдуманы/некорректны с технической точки зрения. |

### Критерий 2: Полнота покрытия остальных разделов – 1 балл
| Оценка | Дескрипторы (что наблюдает преподаватель) |
| :--- | :--- |
| **1 балл** | Четко и кратко описано назначение команды. Названы и осмысленно (не просто списком) разобраны 2–3 ключевых параметра. Упомянуты важные особенности поведения или риски (например, переписывание истории при `rebase`). Соблюден регламент времени (5–7 мин). |
| **0 баллов** | Отсутствует описание параметров или особенностей поведения. Доклад представляет собой бессвязное чтение общей теории без привязки к конкретной команде. Существенное превышение лимита времени, срывающее график занятия. |

---

## 4. Роль преподавателя в модерации дискуссии

Поскольку студент мог использовать ИИ для подготовки, основная проверка глубины знаний происходит на этапе дискуссии (2–3 минуты). Преподавателю рекомендуется использовать следующие приемы:

1. **Проверка на понимание через «Что, если...»:** 
   - *«Что произойдет с этой командой, если в репозитории есть незакоммиченные изменения?»*
   - *«Как эта команда повлияет на коллег, если вы уже отправили (push) изменения в удаленный репозиторий?»*
2. **Запрос альтернативы:** 
   - *«Вы рассказали про `git reset`. А в каком случае вместо него лучше использовать `git revert` и почему?»*
3. **Вовлечение группы:** 
   - Предложите аудитории задать один уточняющий вопрос или дополнить доклад примером из своего опыта. Это развивает культуру код-ревью и командной коммуникации.

## 5. Примерные темы: на что обратить внимание

| Тема доклада | Ключевые команды | Ожидаемые акценты в Use Cases (на что смотреть при оценке) |
| :--- | :--- | :--- |
| **Линеаризация истории кода** | `git rebase` | Обновление своей ветки перед merge request, чтобы история была прямой и чистой. Риск: нельзя делать на публичных ветках. |
| **Идеализация истории кода** | `git rebase -i` | Объединение (squash) промежуточных коммитов ("wip", "fix typo") перед отправкой кода на ревью. Изменение порядка коммитов. |
| **Копирование и «припрятывание» изменений** | `git stash`, `git cherry-pick` | `stash`: срочное переключение на другую задачу (hotfix) без коммита черновика. `cherry-pick`: перенос конкретного исправления из одной ветки в другую (например, из `develop` в `release`). |
| **Отслеживание ошибок и авторов** | `git bisect`, `git annotate` / `git blame` | `bisect`: бинарный поиск коммита, который сломал сборку или тесты. `blame`: выяснение контекста и автора изменения конкретной строки перед её рефакторингом. |
| **Подходы к удалению изменений** | `git revert`, `git reset` | `revert`: безопасная отмена уже отправленного в общий репозиторий коммита (создает новый коммит). `reset`: локальная отмена ошибок до `push` (soft, mixed, hard). |

---

## 6. Рекомендации по фиксации результатов

1. Оценивание производится непосредственно в ходе занятия с использованием ведомости или электронной системы оценивания.
2. При выставлении **0 баллов** по любому из критериев преподаватель обязан кратко аргументировать оценку в ходе дискуссии, указав студенту на конкретный недостающий элемент (например: *«Вы не привели ни одного практического сценария, кроме базового, поэтому первый критерий не выполнен»*).
3. Допускается начисление баллов за активное и содержательное участие в дискуссии в качестве оппонента (по усмотрению преподавателя и в рамках общей балльно-рейтинговой системы дисциплины), что стимулирует вовлеченность всей группы.
