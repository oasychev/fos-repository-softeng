# КИМ-4.2. [Лабораторная работа]

Подготовительная часть лабораторных работ проводится в аудитории. Финализация проекта проводится студентом самостоятельно. Защита работы включает демонстрацию работы каждого задания из аудиторной и самостоятельной части работы, а также ответы на теоретические и практические вопросы. 

## Варианты лабораторных работ:

**Вариант 1. Консольный анализатор погодных данных**

*Аудиторная часть (4 часа)*

Цель: Разработать консольное приложение для получения, валидации и отображения данных о погоде из публичного API (например, OpenWeatherMap https://openweathermap.org/).

Задания (работа в аудитории):

1 Инициализация проекта (Poetry): Создать проект weather_analyzer через poetry new. Добавить зависимости: requests, pydantic, pydantic-settings, python-dotenv. Добавить dev-зависимости: pytest, pytest-cov, pytest-mock, pre-commit.

2 Структура и код: Создать модули: models.py — Pydantic-модель WeatherRecord с полями: city, temperature, humidity, pressure, timestamp. fetcher.py — функция fetch_weather(city: str) -> dict, выполняющая GET-запрос к API. config.py — класс Settings с переменными: API_KEY, BASE_URL, LOG_LEVEL, TIMEOUT. cli.py — точка входа с argparse для приёма названия города.

3 Качество кода: Добавить type hints для всех функций. Написать docstrings в Google-стиле. Настроить pre-commit хуки: trailing-whitespace, end-of-file-fixer, ruff.

4 Логирование: Настроить логгер с выводом в консоль и файл weather.log. Логировать начало и завершение запроса, ошибки.
Результат к концу занятия: Приложение, которое по команде python -m weather_analyzer.cli Moscow выводит в консоль текущую погоду.

*Самостоятельная работа (продолжение проекта)*

1 Написать модульные тесты с использованием pytest и pytest-mock: Успешное получение данных. Обработка ошибок (неверный город, таймаут). Проверка валидации модели (некорректные значения).

2 Добавить поддержку .env и проверить работу приложения с разными уровнями логирования.

3 Добиться покрытия тестами ≥85% (pytest --cov).

4 Интегрировать запуск тестов в pre-commit.

5 Подготовить README.md с инструкцией по установке и запуску.

**Вариант 2. Парсер и валидатор данных с веб-страниц**

*Аудиторная часть (4 часа)*

Цель: Создать инструмент для извлечения структурированных данных из HTML-страниц (например, https://books.toscrape.com), их валидации и сохранения в JSON.
Задания (работа в аудитории):

1 Инициализация (Poetry): Проект html_parser. Добавить зависимости: requests, beautifulsoup4, pydantic, pydantic-settings, python-dotenv.

2 Структура: Модули: models.py — BookRecord с полями: title, author, price, rating, availability. parser.py — функция parse_page(url: str) -> list[dict], использующая BeautifulSoup. saver.py — функция save_to_json(data: list[BookRecord], filename: str). cli.py — принимает --url и --output.

3 Качество кода: Type hints, docstrings, pre-commit с ruff и flake8.

4 Логирование: Логировать количество найденных книг, ошибки парсинга, процесс сохранения.
Результат к концу занятия: Приложение, сохраняющее данные в JSON-файл.


*Самостоятельная работа (продолжение проекта)*

1 Написать тесты: Мокирование requests.get для тестирования парсера без реального запроса. Проверка валидации модели (некорректные цены, отсутствие автора). Тестирование сохранения в JSON.

2 Добавить обработку пагинации (обход нескольких страниц) и логирование прогресса.

3 Добавить конфигурацию через .env (таймаут, user-agent, базовый URL).

4 Настроить pre-commit для автоматического запуска тестов.


**Вариант 3. Модуль конфигурации для микросервисов с поддержкой множества источников**

*Аудиторная часть (4 часа)*

Цель: Создать гибкий модуль конфигурации, который загружает настройки из .env, JSON-файла и аргументов командной строки с приоритетом (например, https://dummyjson.com/, https://open-meteo.com/, примеры конфигурационных файлов могут создавать студенты самостоятельно).

Задания (работа в аудитории):

1 Инициализация (Poetry): Проект config_manager. Зависимости: pydantic, pydantic-settings, python-dotenv, pyyaml. Dev: pytest.

2 Структура: models.py — Pydantic-модели: DatabaseConfig, LoggingConfig, AppConfig (объединяет все). loader.py — функции load_from_env(), load_from_json(), load_from_yaml(), load_from_cli(). config.py — класс ConfigManager, который объединяет источники с приоритетом (CLI > .env > файл > defaults). cli.py — демонстрационный скрипт, выводящий загруженную конфигурацию.

3 Качество кода: Type hints, docstrings, pre-commit.

4 Логирование: Логировать, откуда взято каждое значение (источник).
Результат к концу занятия: Загрузчик конфигураций.


*Самостоятельная работа (продолжение проекта)*

1 Написать тесты для всех сценариев загрузки: Приоритет источников. Частичное переопределение. Обработка отсутствующих файлов. Валидация типов.

2 Добавить поддержку переменных окружения с префиксом (например, MYAPP_).

3 Создать пример использования в виде небольшого приложения (например, HTTP-клиент с настройками).

4 Добавить поддержку секретов (маркировка полей как secret, скрытие при выводе).

5 Интегрировать pre-commit.

## Критерии оценивания:

Выполнение аудиторной части задания - 0-1 балл

Выполнение самостоятельной части задания - 0-2 балл

Защита работы - 0-4 балла
