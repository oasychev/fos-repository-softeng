# Требования к репозиториям для лабораторной работы


## Часть 1. Внесение изменений в репозиторий для улучшения его качества 

Репозитории для этой части, как правило, содержат один файл программного кода и воспроизводят необходимую ситуацию для выполнения действия.

При этом в репозиториях специально создаются пары конфликтующих коммиты, под которыми понимаются коммиты меняющие одну и ту же строчку кода либо соседние строчки кода. Для удобства проверки ответа конфликтующие изменения должны быть независимыми, то есть допускать выполнение в любом порядке. Конфликтные коммиты следует помечать и показывать преподавателю при получении задания.


### Копирование коммитов между ветками

Репозиторий содержит не менее 3х веток, каждая из которых содержит не менее 3х коммитов. Каждая пара веток должна содержать конфликтные коммиты.

###  Удаление изменений

Репозиторий содержит линейную структуру из 5-7 коммитов. Может содержать конфликтующие пары коммитов, чтобы студенты проработали специфический конфликт при удалении близких изменений.

Удаление может происходить как полное, так и с созданием параллельного коммита.

### Линеаризация репозитория

Репозиторий содержит не менее 3х веток, каждая из которых содержит не менее 3х коммитов. Каждая пара веток должна содержать конфликтные коммиты. 

### Идеализация истории

Задачи этого типа выполняются на репозитории линейной структуры. За один раз может выполняться одна или несколько задач. При выполнении одной задачи, рекомендуется не менее 5 коммитов и одна пара конфликтных, при более 1й задачи, не менее 10 коммитов и две пары конфликтных.

#### Перемещение коммитов

Перемещение обычно делается так чтобы пара конфликтующих коммитов поменялась местами, чтобы студенты проработали разрешение специфического конфликта при перестановке изменений.

На практике, эта задача часто совмещается со следующей (переместить коммит к тому, который он исправляет, и слить с ним).

#### Слияние коммитов

Не требует дополнительных условий.

#### Разбиение коммитов.

Коммит для разбиения должен менять несколько строчек. Эти изменения могут быть в разных ханках (легче) или в одном ханке (сложнее).

## Часть 2. Анализ репозитория

Для этой части необходимы репозитории с запускаемым кодом, предпочтительно с небольшим количеством разветвлений и слияний. Код должен компилироваться и собираться (если это компилируемый язык программирования) и запускаться на компьютерах на которых работают студенты. Знакомство с языком программирования на котором код написан не обязательно кроме умения собрать и запустить проект.

Заранее в репозитории, с помощью той же команды bisect, необходимо найти "первый запускаемый коммит" (ПЗК), который позволяет запустить программу и посмотреть на ее начальное состояние. Между ПЗК и концом репозитория должно быть не менее 250 коммитов -- предпочтительно 1000 и более. В связи с логарифмической производительностью команды bisect, большие репозитории не являются проблемой, но следует избегать очень большого количества слитых веток которые замедляют бисекцию.

Программа должна иметь не менее 7-10 функций, чье поведение значительно изменилось между ПЗК и последним коммитом. Преподаватель выдает задание на основе этой разницы, поэтому очень желательно иметь возможность запустить две версии программы на одном компьютере (что может дисквалифицировать некоторые серверные программы). С консольными программами работать, как правило, неудобно из-за отсутствия ясно видимого интерфейса, позволяющего быстро идентифицировать изменения в функциональности программы.
