# КИМ-3.2. [Лабораторная работа]

## Часть 1. Внесение изменений в репозиторий для улучшения его качества 

Внесение изменений может включать по отдельности или вместе следующие изменения: 
1. копирование коммитов между ветками, 
2. удаление изменений, 
3. линеаризацию репозитория, 
4. идеализацию истории: 
    1. перемещение коммитов (в том числе через конфликтующие коммиты),
	2. слияние коммитов
	3. разбиение коммитов.
	
При этом запланированы типовые конфликты при выполнении действий и студенты должны их разрешить.

Базовые репозитории для этой части разрабатывают студенты на основе [требований](kim-02-requirements.md), при этом при выполнении студенты обмениваются репозиториями и выполняют работу по репозиторию другого человека.

Критерии оценивания:
* корректность выбора комманд Git и определения их параметров - 0-1 балл
* корректность разрешения конфликтов - 0-2 балла
* время выполнения работы 0-1 балл.

Данная часть лабораторной работы проводится в два этапа:
1. Подготовка репозиториев и тренировка. Студенты во время самостоятельной работы готовят тренировочные репозитории в соответствии с требованиями для выполнения различных изменений и тренируются на них.
2. На лабораторной работе студенты обмениваются репозиториями (выбранными с помощью генератора случайных чисел) и выполняют задания на репозиториях друг друга.

После выполнения работы, преподаватель проверяет корректность сделанных изменений в репозитории.

## Часть 2. Анализ репозитория

Проведение бисекции для нахождения коммита внесшего определенные изменения. Для этой части необходим репозиторий содержащий запускаемый программный код, который может браться из Open Source проектов или предоставляться индустриальными партнерами.

Поскольку поиск конкретно ошибок в промышленных репозиториях затруднен тем, что ошибок сравнительно немного и их исправления нередко задокументированы в коде, то студентам предлагается вместо ошибки найти коммит, реализовавший определенную функционалность (в ее результирующем виде) с помощью команды bisect.

Преподаватель выдает задание основываясь на видимой разнице функциональности программы в состоянии первого запускаемого коммита и последнего коммита в репозитории.

Критерии оценивания:
* время выполнения работы 0-2 балла;
* корректность использования команды 0-1 балл.
