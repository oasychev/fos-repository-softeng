# Курсовая работа по дисциплине «Разработка прикладных программных решений для систем искусственного интеллекта»

---

## Общее описание проекта

**Тема курсовой работы:** Разработка полнофункционального веб-сервиса для прогнозирования оттока клиентов (Churn Prediction) телеком-компании с использованием методов машинного обучения, REST API и контейнеризации.

**Цель работы:** Спроектировать, разработать и развернуть сквозное ИИ-приложение, охватывающее все этапы жизненного цикла: от анализа и обработки данных до развертывания модели в виде микросервиса с мониторингом.

**Исходные данные:** Датасет телеком-компании с историческими данными о клиентах (Telco Customer Churn Dataset с Kaggle или аналогичный открытый датасет).

**Технологический стек:**
- Python 3.10+
- Poetry — управление зависимостями
- Pandas, NumPy — анализ и обработка данных
- Matplotlib, Seaborn — визуализация
- Scikit-learn — машинное обучение
- Pydantic, python-dotenv — конфигурация и валидация
- FastAPI — REST API
- PostgreSQL — база данных
- Docker, Docker Compose — контейнеризация
- Git, pre-commit, pytest — инженерные практики

# Этап 2. Разработка ML-модели и пайплайна

## Цель этапа
Разработать модель машинного обучения для прогнозирования оттока клиентов, построить ETL-пайплайн для загрузки данных в БД и оценить качество модели.

## Задачи

### 2.1. ETL-пайплайн (PostgreSQL + Python)
- Создать базу данных в PostgreSQL (локально или в Docker-контейнере).
- Разработать скрипт для загрузки данных из исходного CSV-файла в нормализованную БД.
- Использовать `psycopg2` или `asyncpg` для подключения к БД.
- Реализовать пакетную вставку данных с использованием `COPY` для оптимизации.
- Написать SQL-запросы для извлечения агрегированных данных (например, количество услуг на клиента, средний платёж и т.д.) с использованием `GROUP BY`, `JOIN`, оконных функций.

### 2.2. Разработка ML-модели (Scikit-learn)
- Разделить данные на обучающую и тестовую выборки (70/30 или 80/20) с учётом стратификации.
- Обучить **минимум 3 модели** для решения задачи бинарной классификации:
  - Логистическая регрессия (базовая модель).
  - Случайный лес (Random Forest).
  - Градиентный бустинг (XGBoost, LightGBM или CatBoost).
- Провести подбор гиперпараметров (GridSearchCV или RandomizedSearchCV).
- Выполнить оценку качества моделей с использованием метрик: точность (Accuracy), полнота (Recall), точность (Precision), F1-мера, ROC-AUC.
- Сравнить модели между собой и выбрать лучшую.

### 2.3. Построение пайплайна предобработки
- Создать полноценный пайплайн с использованием `sklearn.pipeline.Pipeline`, включающий:
  - Стандартизацию числовых признаков.
  - Кодирование категориальных признаков.
  - Выбор лучших признаков (опционально).
- Сохранить обученный пайплайн и модель с помощью `joblib` или `pickle`.

### 2.4. Логирование и тестирование
- Настроить логирование (модуль `logging`) всех этапов обучения и обработки данных.
- Написать модульные тесты (pytest) для:
  - Проверки работы ETL-скриптов.
  - Проверки загрузки и валидации данных.
  - Проверки пайплайна предобработки.
- Добиться покрытия тестами ≥80%.

### 2.5. Документирование и инженерные практики
- Продолжить использование `type hints` и `docstrings`.
- Интегрировать запуск тестов в `pre-commit`.
- Добавить конфигурацию через `.env` (параметры подключения к БД, пути к данным).

## Результат этапа
- Обученная ML-модель и пайплайн предобработки (сохранённые в файлы).
- ETL-скрипты для загрузки данных в PostgreSQL.
- Отчёт с результатами сравнения моделей и выбранными гиперпараметрами.

# Этап 3. Разработка REST API и интеграция с БД

## Цель этапа
Разработать веб-сервис на FastAPI с эндпоинтами для получения прогнозов, управления данными и получения метрик качества, с интеграцией с PostgreSQL и загруженной ML-моделью.

## Задачи

### 3.1. Разработка API на FastAPI
- Создать приложение FastAPI с базовой структурой (роутеры, схемы Pydantic).
- Реализовать **минимум 4 эндпоинта**:

| Эндпоинт | Метод | Назначение |
|----------|-------|------------|
| `/predict` | POST | Принимает данные о клиенте (в формате JSON), возвращает прогноз оттока (0/1) и вероятность. |
| `/health` | GET | Проверка работоспособности сервиса. |
| `/metrics` | GET | Возвращает метрики качества модели (Accuracy, F1, ROC-AUC) на тестовых данных. |
| `/features` | GET | Возвращает список признаков, используемых моделью, с описанием. |
| `/history` | GET (опционально) | Возвращает историю прогнозов (с пагинацией). |

### 3.2. Валидация данных (Pydantic)
- Создать Pydantic-схемы для валидации входных данных на эндпоинте `/predict`.
- Добавить валидацию типов, диапазонов значений и обязательных полей.
- Обрабатывать ошибки валидации и возвращать понятные сообщения пользователю.

### 3.3. Асинхронная работа с БД
- Использовать `asyncpg` для асинхронного подключения к PostgreSQL.
- Реализовать асинхронный пул соединений.
- Написать асинхронные функции для:
  - Сохранения запросов и ответов в таблицу `predictions_history`.
  - Получения статистики по истории прогнозов.
  - Получения данных о клиентах из БД при необходимости.

### 3.4. Логирование и обработка ошибок
- Настроить логирование запросов, ошибок и времени выполнения эндпоинтов.
- Реализовать кастомные исключения и глобальный обработчик ошибок.
- Логировать каждое обращение к `/predict` с идентификатором запроса.

### 3.5. Документация API
- Использовать встроенную документацию Swagger (`/docs`) и ReDoc (`/redoc`).
- Добавить описания для всех эндпоинтов, примеры запросов и ответов.
- Подготовить OpenAPI-спецификацию в формате JSON.

### 3.6. Тестирование
- Написать интеграционные тесты для эндпоинтов с использованием `pytest` и `httpx`.
- Написать тесты для асинхронных запросов к БД.
- Использовать `pytest-mock` для мокирования внешних зависимостей (например, ML-модели).
- Добиться покрытия кода API ≥85%.

## Результат этапа
- FastAPI-приложение с реализованными эндпоинтами.
- Pydantic-схемы для валидации.
- Асинхронное подключение к PostgreSQL.
- Swagger-документация API.
- Тесты и логи.

# Этап 4. Контейнеризация, деплой и мониторинг

## Цель этапа
Контейнеризовать разработанное приложение с использованием Docker, настроить многоконтейнерное развертывание через Docker Compose и организовать непрерывную интеграцию через GitHub Actions.

## Задачи

### 4.1. Контейнеризация приложения (Docker)
- Создать `Dockerfile` для FastAPI-приложения:
  - Использовать официальный Python-образ (python:3.10-slim).
  - Установить зависимости из Poetry (`poetry install`).
  - Скопировать код и модель.
  - Определить команду запуска (`CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]`).
- Создать `Dockerfile` для PostgreSQL (или использовать готовый образ).
- Оптимизировать размер образа (многослойная сборка, использование `--no-cache-dir`).

### 4.2. Многоконтейнерное развертывание (Docker Compose)
- Создать `docker-compose.yml` для запуска трёх сервисов:
  - **app** — FastAPI-приложение.
  - **db** — PostgreSQL.
  - **pgadmin** (опционально) — для управления БД.
- Настроить переменные окружения через `.env` для всех сервисов.
- Настроить сети и тома (volumes) для сохранения данных БД.
- Убедиться, что приложение ожидает запуска БД (использовать `depends_on` или скрипт `wait-for-it.sh`).

### 4.3. CI/CD через GitHub Actions
- Настроить GitHub Actions для автоматической сборки и тестирования приложения при каждом push-е:
  - Проверка форматирования (ruff).
  - Запуск тестов (pytest) в контейнере.
  - Сборка Docker-образа.
  - Публикация образа в Docker Hub (или GitHub Container Registry).
- Создать пайплайн для автоматического деплоя на сервер (опционально).

### 4.4. Мониторинг и наблюдаемость
- Интегрировать логирование в JSON-формате (для совместимости с ELK или Loki).
- Добавить эндпоинт `/metrics` с Prometheus-метриками (количество запросов, время ответа, ошибки) — использовать `prometheus-client`.
- Настроить сбор метрик и визуализацию в Grafana (опционально, для продвинутого уровня).

### 4.5. Документирование и финальная проверка

  - Инструкции по установке через Poetry.
  - Инструкции по запуску через Docker Compose.
